# FOM

JVM-библиотека для управления **долгоживущими процессами**, организованными в
направленный ациклический граф зависимостей. Каждый процесс один раз
инициализирует своё состояние (возможно, дорогое вычисление) и сохраняет его в
append-only лог, так что перезапуск JVM восстанавливает состояние **без
повторного вычисления**.

```java
var graph = new GraphBuilder()
    .add("Inventory", InventoryInit::new, InventoryInit::new)
        .handles(GetInventoryModel.class)
    .add("Products", ProductsInit::new, ProductsInit::new, "Inventory")
        .handles(GetProductModel.class)
    .build();

try (var backend = new LocalFileLogBackend(Path.of("/var/lib/fom/log.bin"));
     var engine = new Engine(EngineConfig.defaults(), backend, new FurySerDe(), true)) {
    engine.newGraph(graph);

    var pm = (ProductModel) engine.query(new GetProductModel("PUB1"))
        .toCompletableFuture().get(5, TimeUnit.SECONDS);
}
```

## Какую задачу это решает?

Часть состояния приложения **дорого строить, но дёшево хранить**: поисковый
индекс, денормализованная read-модель, прогретый из многих источников кэш.
Перестраивать его на каждый деплой или рестарт расточительно — иногда
непозволительно.

FOM моделирует каждый такой кусок состояния как **процесс** с двумя фазами:

- **`init`** — вычислить состояние один раз и записать результат в лог как набор
  байтовых «ячеек свойств».
- **`load`** — восстановить живой, отвечающий на запросы объект из этих ячеек.

При холодном старте процесс выполняет `init`, затем `load`. При **тёплом
рестарте** против того же лога движок видит, что состояние уже было
инициализировано, и выполняет только `load` — это и есть цель *идемпотентного
рестарта*. См. [Идемпотентный рестарт](concepts/idempotent-restart.md).

Процессы образуют **граф**: потребитель может зависеть от поставщиков и
обращаться к ним во время обслуживания. Когда состояние поставщика меняется,
*реактивные* потребители автоматически переинициализируются в порядке
зависимостей. См. [Граф и маршрутизация](concepts/graph-and-routing.md) и
[Реактивный каскад](concepts/reactive-cascade.md).

## Принципы проектирования

- **Лог — источник истины.** Все долговечные изменения состояния добавляются в
  [append-only лог](concepts/the-log.md); состояние движка в памяти — это
  проекция, которую всегда можно перестроить воспроизведением лога.
- **Один писатель за раз.** Добавлять записи может только один *лидер*; лидерство
  обеспечивается бэкендом (файловая блокировка или advisory-lock в Postgres). См.
  [Несколько узлов](guides/multi-node.md).
- **Идиоматичное ядро с минимумом зависимостей.** `fom-core` — это чистый **Java
  21 + Project Loom** с **нулём не-JDK зависимостей** во время выполнения, кроме
  `slf4j-api`. Каждая интеграция (Kotlin DSL, Fury, Spring, Micrometer …) — это
  отдельный опциональный модуль.
- **Изоляция в стиле акторов.** Каждым процессом управляет один диспетчер на
  виртуальном потоке, читающий мейлбокс; пользовательский `compute` выполняется
  на собственном виртуальном потоке, поэтому диспетчер никогда не блокируется.
  См. [Жизненный цикл процесса](concepts/process-lifecycle.md).
