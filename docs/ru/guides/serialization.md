# Сериализация

Движку нужно превращать пользовательские значения и граф в байты для лога и
обратно. Это работа SPI **`SerDe`**. Выбор и усиление `SerDe` — единственное
самое важное для безопасности решение конфигурации; читайте эту страницу вместе с
[Безопасностью](../security.md).

## SPI `SerDe`

```java
public interface SerDe {
    byte[] serializeTriggerValue(String processName, Object value);
    Object loadTriggerValue(String processName, byte[] bytes);
    byte[] serializeProcessProperties(Map<String, byte[]> properties);
    Map<String, byte[]> loadProcessProperties(byte[] bytes);
    byte[] serializeProcessGraph(Graph graph);
    Graph loadProcessGraph(byte[] bytes);
}
```

Он сериализует три вещи: **значения триггеров** (произвольные пользовательские
объекты), **свойства процесса** (`Map<String, byte[]>`) и **граф** (который
встраивает ваши фабрики-суплаеры и любые резолверы динамических маршрутов — значит,
они должны быть сериализуемы).

`SerDe` — на весь движок. Он отличается от `Codec` на отдельное значение,
используемого `TypedKey` внутри ячеек свойств (см.
[Конфигурацию](configuration.md#типизированные-ячейки-свойств)).

## Выбор реализации

### `FurySerDe` (рекомендуется для продакшена)

`fom-fury` оборачивает [Apache Fury](https://fury.apache.org/): компактный,
быстрый, дружественный к эволюции схемы (добавление nullable-поля в record не
сломает читателей, собранных против старой схемы). Создавайте один экземпляр на
движок и переиспользуйте.

```java
import io.fom.fury.FurySerDe;

var serDe = new FurySerDe();                 // разрешающий (по умолчанию)
var strict = new FurySerDe(true);            // requireClassRegistration = true
```

- Конструктор без аргументов оставляет регистрацию классов **выключенной** — без
  настройки, но Fury инстанцирует любой класс, названный в байтах.
- `new FurySerDe(true)` включает гейт регистрации классов Fury — его сильнейшую
  защиту от вредоносных байтов. Тогда вы контролируете, какие именно классы можно
  десериализовать. Это opt-in, потому что смена дефолта сломала бы сериализацию
  произвольных пользовательских значений триггеров.

Когда выбирать строгий режим — см. [Безопасность](../security.md).

### `JavaSerializableSerDe` (запасной, тесты, быстрый старт)

Встроенный `SerDe` `fom-core` использует `ObjectOutputStream`/`ObjectInputStream`.
Все значения должны реализовывать `java.io.Serializable`.

```java
import io.fom.serde.JavaSerializableSerDe;
import io.fom.serde.ObjectInputFilters;

var serDe = new JavaSerializableSerDe();                      // фильтр ограничения ресурсов
var hardened = new JavaSerializableSerDe(myObjectInputFilter); // ваш allowlist
```

Каждый `readObject()` выполняется под `ObjectInputFilter`:

- по умолчанию (`ObjectInputFilters.resourceLimits()`) ограничивает глубину,
  ссылки, байты потока и длину массива, чтобы притупить «бомбы десериализации», при
  этом допуская неизвестные пользовательские классы;
- передайте свой `ObjectInputFilter` (например, строгий allowlist классов) через
  конструктор для продакшена, или задайте общесистемный `jdk.serialFilter`.

!!! danger "Java-сериализация — известный вектор RCE"
    Даже с фильтром ограничения ресурсов нефильтрованная десериализация классов из
    байтов, контролируемых атакующим, опасна. Для продакшена с недоверенными
    байтами лога предпочитайте `FurySerDe` с `requireClassRegistration(true)` или
    передавайте строгий allowlist-фильтр. См. [Безопасность](../security.md).

## Усиление полезной нагрузки лога (автоматически)

**Бэкенды лога** (`LocalFileLogBackend`, `PostgresLogBackend`) восстанавливают
только собственные `io.fom.*` записи событий плюс JDK-типы и `byte[]` — ваши
объекты остаются внутри ячеек `byte[]`, пока `SerDe` не декодирует их отдельно.
Поэтому эти бэкенды применяют **строгий allowlist**-фильтр
(`ObjectInputFilters.logPayload()`) на своём внутреннем `readObject`, отклоняя
любой класс не из `io.fom`/JDK. Это встроено; настраивать не нужно.

!!! note "CRC — это не MAC"
    Файловый бэкенд обрамляет каждое событие CRC32 для обнаружения *повреждений*.
    Это не защищённая от подделки подпись — относитесь к файлу лога как к локально
    владеемому артефакту с контролем доступа.

## Совместимость между бэкендами

Все три встроенных бэкенда внутренне сериализуют полезные нагрузки событий
Java-сериализацией, поэтому лог, записанный одним, читается другим. **Выбранный
вами `SerDe`** влияет на то, как кодируются *ваши* байты триггера/свойств/графа
внутри этих событий — держите его согласованным между рестартами одного лога.
