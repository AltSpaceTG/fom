# Несколько узлов и лидерство

FOM — single-writer: в любой момент один **лидер** добавляет в лог; остальные
экземпляры — фолловеры. Запуск более одного экземпляра против одного лога требует
бэкенда, арбитрирующего лидерство между процессами — это `PostgresLogBackend`.

## Модель лидерства

- Последнее событие `LogLeader` называет текущего лидера. Только этот `instanceId`
  может `append`/`compact`; записи устаревшего лидера возвращают
  `Optional.empty()`. См. [Лог](../concepts/the-log.md#лидерство-один-писатель).
- Поверх правила лога бэкенд не даёт двум экземплярам *обоим* считать себя
  лидером:
    - `LocalFileLogBackend` — эксклюзивная ОС-блокировка соседнего файла `.lock`
      (только одна машина).
    - `PostgresLogBackend` — **advisory-lock** Postgres по имени таблицы,
      удерживаемый на выделенном соединении до `close()`.

## Настройка Postgres

```java
import io.fom.jdbc.PostgresLogBackend;

// На каждом узле, указывая на ту же БД + logId:
var backend = new PostgresLogBackend(dataSource, "orders");
var engine  = new Engine(EngineConfig.defaults(), backend, new FurySerDe(), true);
engine.newGraph(graph);
```

- **Первый** узел, сконструировавший бэкенд, берёт advisory-lock и становится
  лидером; восстанавливает состояние и обслуживает.
- **Второй** узел, конструирующий `PostgresLogBackend(dataSource, "orders")`,
  быстро падает с `IllegalStateException` («advisory lock») — лидером он не стал.
  Поймайте это и либо повторите позже, либо работайте как standby, конструирующий
  бэкенд только после выхода лидера.

```java
PostgresLogBackend backend;
try {
    backend = new PostgresLogBackend(dataSource, "orders");
} catch (IllegalStateException leaderHeldElsewhere) {
    // лидер — другой узел; отступите и повторите, или встаньте в standby.
    return;
}
```

## Failover

Когда процесс лидера выходит (или его соединение рвётся), Postgres освобождает
advisory-lock. Standby, тогда конструирующий бэкенд, берёт блокировку, становится
лидером, восстанавливается из лога и продолжает — [идемпотентный
рестарт](../concepts/idempotent-restart.md) на другом узле.

!!! note "Восстановление — работа лидера"
    Снапшоты/компактизация и установка графа — только для лидера. Фолловер,
    ставший лидером, воспроизводит лог для пересборки состояния; процессы тёпло
    загружают свой последний живой `LogInitialized`.

## Выбор бэкенда

| У вас… | Используйте |
|---|---|
| один процесс, эфемерное состояние | `InMemoryLogBackend` |
| один процесс, долговечное состояние | `LocalFileLogBackend` |
| несколько процессов/узлов, HA | `PostgresLogBackend` |

Ключ advisory-lock — 64-битный хеш имени таблицы, поэтому разные `logId` не
коллизируют по блокировке.

## Замечание о безопасности

Любой, кто может писать в таблицу лога Postgres, может подложить байты, которые
лидер прочитает. Десериализация бэкенда фильтруется allowlist'ом, но всё равно
ограничьте привилегии таблицы. См. [Безопасность](../security.md).
