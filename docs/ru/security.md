# Безопасность

FOM сохраняет состояние, сериализуя объекты в лог и читая их обратно. Чувствительна
сторона чтения: **десериализация байтов, контролируемых атакующим, — классический
вектор удалённого выполнения кода (RCE).** Эта страница описывает модель угроз, что
библиотека делает по умолчанию, и что нужно сделать вам для усиленного
развёртывания.

## Модель угроз

Логу доверяют ровно настолько, насколько доверяют его **хранилищу**:

- **`LocalFileLogBackend`** — любой, кто может писать в файл лога (или его
  каталог), может подложить байты, которые лидер десериализует при следующем
  старте/воспроизведении.
- **`PostgresLogBackend`** — любой, кто может `INSERT`/`UPDATE` в таблицу лога
  (скомпрометированный аккаунт БД, SQL-инъекция в другом месте, вредоносный
  co-tenant), может сделать то же.
- **CLI `fom-log`** — `inspect`/`diagnose` десериализуют любой переданный путь.

CRC32 в формате файла обнаруживает **повреждение**, а не подделку — это не MAC, и
атакующий, переписывающий файл, просто пересчитывает его.

## Что библиотека делает по умолчанию

- **Десериализация полезной нагрузки лога фильтруется allowlist'ом.** Файловый и
  Postgres-бэкенды восстанавливают только собственные `io.fom.*` записи событий
  плюс JDK-типы и `byte[]` — ваши объекты остаются внутри ячеек `byte[]`, пока ваш
  `SerDe` не декодирует их отдельно. Эти бэкенды устанавливают
  `ObjectInputFilters.logPayload()` на внутренний `readObject`, который
  **отклоняет любой класс не из `io.fom`/JDK** и ограничивает глубину/ссылки/размер.
  Это автоматически и не настраивается.
- **`JavaSerializableSerDe` работает под фильтром ограничения ресурсов.**
  Поскольку он должен принимать произвольные *пользовательские* значения
  триггеров/свойств, он не может применять allowlist классов; по умолчанию
  применяет `ObjectInputFilters.resourceLimits()` (ограничения глубины, ссылок,
  байтов потока, длины массива), чтобы притупить «бомбы десериализации».
- **Проверки типа при декодировании.** Каждый результат внутреннего `readObject`
  проверяется через `instanceof` перед использованием, поэтому валидная, но
  неверная по типу нагрузка падает чисто, а не бросает сырой `ClassCastException`.

## Что нужно сделать для продакшена

!!! danger "Усильте сериализатор для недоверенных байтов"
    Если кто-либо, кроме вашего лидера, может писать в хранилище лога, сделайте
    **одно** из:

- Используйте **`FurySerDe` с включённой регистрацией классов**:
  `new FurySerDe(true)` — тогда Fury откажется инстанцировать любой
  незарегистрированный класс. Это сильнейший, рекомендуемый вариант.
- Или передайте **строгий allowlist**-фильтр Java-сериализатору:
  `new JavaSerializableSerDe(myObjectInputFilter)`, где `myObjectInputFilter`
  допускает только ваши известные типы — или задайте общесистемный
  `jdk.serialFilter`.

- **Ограничьте хранилище.** Относитесь к файлу лога как к локально владеемому
  артефакту с контролем доступа (строгие права файла). Для Postgres ограничьте
  привилегии таблицы только ролью приложения; не делите таблицу с недоверенными
  тенантами.
- **Запускайте CLI только на доверенных файлах.** `fom-log inspect/diagnose`
  десериализуют нагрузки (под allowlist-фильтром, но всё же) — не направляйте на
  файлы, которые вы не контролируете.
- **Валидируйте недоверенный конфиг.** Cron-строка HOCON парсится `cron-utils`;
  не передавайте cron из недоверенных источников без валидации.
- **Валидируйте недоверенные пути.** `LocalFileLogBackend` принимает `Path` от вас
  как доверенный ввод — если вы когда-либо выводите его из внешнего ввода,
  валидируйте/канонизируйте сами.

## Изоляция тенантов

Обёртка `fom-tenant` — **fail-closed**: её `authzPolicy` по умолчанию запрещает
всё, `query(caller, msg)` отклоняет не-`Routable` сообщения (чей целевой тенант
нельзя узнать до диспетчеризации), а `shutdownTenant` требует авторизованного
вызывающего. Это defence-in-depth *поверх* вашей собственной аутентификации — она
решает, *какой тенант* может трогать вызывающий, а не *кто* он. См.
[Мультитенантность](guides/multi-tenancy.md).

## Чек-лист продакшен-развёртывания

- [ ] **Сериализатор:** `FurySerDe(true)` (зарегистрированные классы) **или**
      строгий `ObjectInputFilter` на `JavaSerializableSerDe` — никогда разрешающий
      дефолт против недоверенных байтов.
- [ ] **ACL хранилища:** права файла лога / гранты таблицы Postgres ограничены
      приложением.
- [ ] **Authz тенантов:** задана явная `authzPolicy` (дефолт запрещает всё,
      поэтому незаданная политика безопасна, но бесполезна).
- [ ] **Входной конфиг:** cron и любой внешний конфиг провалидированы.
- [ ] **Гигиена CLI:** операторы запускают `fom-log` только на доверенных файлах.
- [ ] **Зависимости:** держите `fom-fury`, `postgresql`, `cron-utils` и т. д.
      пропатченными (например, OWASP dependency check в CI).

!!! note
    Пока это не сделано, не запускайте FOM против недоверенного хранилища лога или
    в multi-tenant продакшене.
